synchronized 和 ReentrantLock 的区别

前言

多线程编程中，锁是控制并发访问的核心手段。Java 中最常用的两把锁就是 synchronized 和 ReentrantLock，它们有什么区别？什么时候该用哪个？

核心区别详解

1. 底层实现

synchronized 是 JVM 内置锁，编译后会在字节码中加入 monitorenter 和 monitorexit 指令，JVM 会自动进行锁升级：无锁 → 偏向锁 → 轻量级锁 → 重量级锁。

ReentrantLock 底层依赖于 AQS（AbstractQueuedSynchronizer），通过 CAS + Park/Unpark 实现。

2. 公平锁

synchronized 是非公平锁，线程抢锁完全靠运气。

ReentrantLock 可以选择公平或非公平：

  • 公平锁 — 按等待时间顺序获取锁
  • 非公平锁 — 允许插队，性能更高

3. 等待中断

ReentrantLock 支持 lockInterruptibly()，线程在等待锁的过程中可以被中断。synchronized 不支持。

4. 条件变量

ReentrantLock 可以绑定多个 Condition，实现精准唤醒：

synchronized 只能配合 Object 的 wait/notify，功能有限。

如何选择？

  • 简单同步，用 synchronized — 代码更简洁，JVM 内置无需担心释放问题
  • 需要高级特性（公平锁、尝试加锁、中断等待），用 ReentrantLock
  • 分布式环境下的锁，用 Redis 实现的分布式锁，不依赖 JVM
