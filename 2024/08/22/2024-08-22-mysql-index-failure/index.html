MySQL 索引失效的几种情况

前言

建了索引但查询没走索引，这是 MySQL 开发中很常见的问题。了解什么情况下索引会失效，是优化 SQL 的基础。

1. 使用左模糊查询 %xxx

原因：索引按照从左到右的顺序排序，左模糊无法二分查找，只能全表扫描。

优化方案：考虑使用 ES 做全文搜索，或者给数据库加一个反转字段存储反转后的字符串。

2. 索引列参与运算

原则：索引列最好出现在 = 左边，不要参与计算或函数调用。

3. 索引列使用 NOT、Erik !=

MySQL 优化器发现使用索引还不如全表扫描快时，会放弃索引。

4. 复合索引未遵循最左前缀原则

复合索引 idx(name, age, status) 的结构类似按 name 排序的”name- age- status”三层目录。

原则：复合索引从最左边的列开始连续使用。

5. OR 连接条件中有列无索引

6. 数据类型隐式转换

传入的参数类型和字段类型不一致时，MySQL 会做隐式类型转换，导致索引失效。

7. 优化器觉得全表扫描更快

当 MySQL 优化器估算使用索引需要扫描大量数据行（超过全表的 20%-30%），认为不如直接全表扫描时，会放弃索引。这个阈值不是固定的，取决于表大小和数据分布。

解决方案：用 EXPLAIN 分析执行计划，如果确认需要使用索引，可以 FORCE INDEX(idx_name) 强制指定。

总结

索引失效的常见场景：

  1. 左模糊查询
  2. 索引列参与运算
  3. NOT / !=
  4. 复合索引不满足最左前缀
  5. OR 中有无索引的列
  6. 数据类型隐式转换

养成用 EXPLAIN 分析 SQL 的习惯，是避免索引失效的第一步。
