MySQL EXPLAIN 执行计划分析

前言

慢查询优化是后端开发中的必备技能。而分析慢 SQL 的第一步，就是学会看 EXPLAIN 的输出。

基本用法

执行计划各字段含义：

1. id — 查询序号

id 越大越先执行；id 相同则由上到下执行。如果 id 为 NULL 表示结果集，不需要执行。

2. select_type — 查询类型

类型 含义
SIMPLE 简单 SELECT，不含子查询或 UNION
PRIMARY 主查询（最外层）
SUBQUERY 子查询
DERIVED 派生表（FROM 中的子查询）
UNION UNION 后的查询
UNION RESULT UNION 结果集

3. type — 访问类型（重要）

从最优到最差排序：

类型 含义
system 表只有一行记录，系统表
const 最多匹配一行，通过主键或唯一索引
eq_ref 使用主键或唯一索引关联查询
ref 使用普通索引等值查询
range 索引范围查询（>, <, BETWEEN, IN）
index 全索引扫描，比 ALL 快
ALL 全表扫描，最差情况

优化目标：至少达到 range 级别，避免 ALL。

4. possible_keys — 可能用到的索引

MySQL 认为可能用到的索引列表，不一定真正使用。

5. key — 实际使用的索引

如果为 NULL，说明没有用索引，需要优化。

6. key_len — 索引长度

越短越好。可以通过 key_len 计算联合索引使用了前几个字段。

7. rows — 估算扫描行数

MySQL 估算需要扫描的行数，不是精确值。这个值越小越好。

8. Extra — 额外信息（重要）

常见值及含义：

值 含义
Using filesort 需要额外排序，无法用索引排序，危险信号
Using temporary 需要临时表，危险信号
Using index 覆盖索引，不用回表，性能好
Using index condition 使用了索引下推
Using where 使用了 WHERE 过滤

Using filesort 和 Using temporary 同时出现时，表示需要优化。

优化思路

  1. type 出现 ALL 或 index → 检查是否有合适的索引
  2. key 为 NULL → 需要建索引
  3. Extra 出现 Using filesort → ORDER BY 字段没有索引或没满足最左前缀
  4. rows 数值过大 → 考虑加更精准的索引条件

总结

EXPLAIN 是分析 SQL 性能最直接的工具。重点关注 type、key、rows、Extra 四个字段，找出性能瓶颈，再针对性地加索引或改写 SQL。
