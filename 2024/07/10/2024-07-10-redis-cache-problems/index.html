Redis 缓存穿透、击穿、雪崩详解

前言

使用 Redis 做缓存时，有三个经典问题必须了解：缓存穿透、缓存击穿、缓存雪崩。它们分别是什么？怎么解决？

1. 缓存穿透

问题：查询一个不存在的数据，缓存查不到，数据库也查不到，请求每次都打到数据库。

危害：恶意攻击者用大量不存在的 key 频繁查询，让数据库压力过大。

解决方案

方案一：参数校验
在接口层做基本的数据校验，过滤掉明显非法 id，比如 id <= 0 的直接 reject。

方案二：缓存空值
把查询结果为空的值也缓存起来，key 是查询条件，value 是空字符串或 null，过期时间设置短一些（比如 60 秒）。

方案三：布隆过滤器（Bloom Filter）
把所有存在的 key 存入布隆过滤器，查询前先判断 key 是否可能存在。布隆过滤器判断不存在的 key 一定不存在，判断存在的 key 可能不存在（假阳性）。适合数据量极大、内存有限的场景。

2. 缓存击穿

问题：某个热点 key 突然过期或失效的瞬间，大量请求同时涌入直接打数据库。

特点：只有某一个热点 key 失效，并发量集中在这一 key 上。

解决方案

方案一：互斥锁（分布式锁）
只有一个线程去查数据库重建缓存，其他线程等待。

方案二：热点数据永不过期
物理上不设置过期时间，过期时间存在 key 对应的 value 中。异步更新缓存时检查是否接近过期。这种方案需要额外逻辑控制，复杂度较高。

3. 缓存雪崩

问题：大量缓存 key 在同一时间过期，或者 Redis 本身宕机，导致大量请求同时打到数据库。

解决方案

方案一：过期时间随机化
给缓存过期时间加一个随机偏移量，避免大量 key 同时过期。

方案二：双层缓存策略
本地缓存（EhCache/Caffeine）+ Redis 缓存。本地缓存作为备份，即使 Redis 过期，本地缓存还能撑一段时间。

方案三：Redis 高可用
搭建 Redis Sentinel（主从 + 哨兵）或 Redis Cluster，避免 Redis 单点故障导致雪崩。

方案四：服务熔级降级
缓存服务不可用时，使用 hystrix/sentinel 等限流组件对数据库进行保护，限制并发数。

方案五：提前演练
在生产环境中提前做缓存失效演练，验证系统的容错能力。

三种问题对比

问题 原因 特点 核心解决思路
穿透 查询不存在的数据 每次都打数据库 空值缓存 / 布隆过滤器
击穿 热点 key 失效 某一个 key 并发高 互斥锁 / 永不过期
雪崩 大量 key 同时失效 大面积请求打数据库 过期时间随机 / 多级缓存
