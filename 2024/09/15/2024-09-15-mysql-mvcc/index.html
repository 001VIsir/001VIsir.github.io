MySQL MVCC 机制详解

前言

MVCC（Multi-Version Concurrency Control，多版本并发控制）是 MySQL InnoDB 引擎实现隔离级别的核心技术。它让数据库在不加锁的情况下实现读写并发，提升系统性能。

事务基础

MySQL 中每条数据其实有两个隐藏字段：

  • trx_id — 最近一次修改这条数据的事务 ID
  • roll_pointer — 指向 undo log 中旧版本的指针

每更新一次，数据就会把旧版本写入 undo log，多个版本通过 roll_pointer 形成一条版本链。

ReadView 机制

事务在读取数据时，会生成一个 ReadView，记录当前活跃事务的 ID 列表（未提交的事务）。

ReadView 包含四个关键字段：

字段 含义
m_ids 当前活跃事务的 ID 列表
min_trx_id 活跃事务中的最小 ID
max_trx_id 创建 ReadView 时最大事务 ID + 1
creator_trx_id 当前事务 ID

不同隔离级别的表现

READ COMMITTED — 每次 SELECT 生成新 ReadView

每次读取数据前都会重新生成 ReadView，所以同一事务内两次读取可能看到不同的数据（其他事务提交的新版本对当前事务可见）。

REPEATABLE READ（默认）— 事务开始时生成 ReadView

ReadView 在事务第一条 SELECT 执行时生成，整个事务内复用同一个 ReadView，所以同一事务内多次读取结果一致。

MVCC + Next-Key Lock 解决幻读

幻读是指同一事务内两次查询结果行数不同。InnoDB 在 REPEATABLE READ 级别下通过 Next-Key Lock（记录锁 + 间隙锁）来防止幻读：

这不仅锁住 id = 10 的记录，还会锁住 (10, +∞) 这个间隙，阻止其他事务在这个范围内插入新记录。

MVCC 的优势

  1. 读写不冲突 — 读操作不需要加锁，写操作也不阻塞读
  2. 提高并发 — 多数场景下无需串行执行
  3. 保证隔离性 — 通过版本判断实现不同隔离级别

总结

MVCC 的核心是维护数据的多个版本，通过 ReadView 判断每个事务能看到哪个版本。理解这个机制，有助于理解 MySQL 的隔离级别和并发控制原理。
