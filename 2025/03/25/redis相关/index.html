1.4 SDS 惰性空间释放（Lazy Free）

SDS 不会立即释放多余内存，而是保留 free 空间供将来使用：

惰性释放的好处：

  • 避免频繁 realloc，减少系统调用
  • 如果字符串再次增长，可能直接用现有空间

1.5 扩容次数验证（均摊复杂度）

虽然最坏情况下一次操作可能触发 O(n) 数据复制，但：

  • 每次扩容至少增加 1MB（大数据时）
  • 扩容后可以连续处理多次追加而不再扩容

均摊分析：n 次追加总代价 ≤ 2n，所以**均摊复杂度是 O(1)**。

5.5 为什么用 SkipList 而不是红黑树

维度 SkipList 红黑树
实现复杂度 简单（几百行） 复杂（插入/删除需要旋转、再平衡）
范围查询 只需在 L0 遍历局部节点 需要中序遍历，需要栈
插入删除 只需修改局部指针 可能触发多次旋转
内存占用 约 1.33 * n（平均每节点 1.33 层） 每个节点 3 个指针（left, right, parent）
遍历方向 双向（SkipList 有 backward 指针） 单向（中序遍历需要父指针）
实现确定性 概率型，最坏情况可能退化 确定型，最坏情况有保证

Redis 选择 SkipList 的关键原因：

  1. 范围查询（ZRANGE）极高效：跳表相邻节点在 L0 上直接相连，遍历局部即可
  2. 实现简单：代码量少，易维护
  3. 内存开销可接受：平均 1.33 个指针，vs 红黑树的 3 个

6.4 Stream 的 Rax 结构

Stream 的消息 ID（如 1527854168987-0）被拆分成两部分：

  1. 毫秒时间戳：高位
  2. 序列号：低位

这些 ID 本身具有公共前缀，适合用 Radix Tree 存储。

为什么用 Radix Tree：

  • 消息 ID 是时间戳序列，天然有公共前缀
  • 相同前缀的 ID 共享节点，内存极省
  • 支持 O(k) 查找（k = ID 长度，平均 ~20 字符）

九、关键设计思想总结

9.1 时间换空间 vs 空间换时间

场景 策略 示例
小数据量 紧凑结构，节省内存 ZipList、IntSet
大数据量 高效结构，提升性能 Dict、SkipList、Quicklist
临界点 自动转换编码 Hash 512 字段阈值

9.2 局部性原理的应用

  • Quicklist：多个 ZipList 用指针串起来，每个 ZipList 内存连续
  • ZipList：所有元素连续存储，CPU 缓存命中率高
  • IntSet：整数数组，CPU 缓存友好

9.3 渐进式策略

  • 渐进式 Rehash：分摊扩容代价，避免阻塞
  • 惰性空间释放：SDS 不立即归还内存，减少系统调用
  • LZF 压缩：按需压缩 Quicklist 中间节点

9.4 多层索引

  • SkipList：L0~L32 多层索引，O(log n) 查询
  • Radix Tree：公共前缀合并，O(k) 查找（k = key 长度）
  • Dict + SkipList：成员 O(1) 查找 + 分值范围 O(log n) 查询
