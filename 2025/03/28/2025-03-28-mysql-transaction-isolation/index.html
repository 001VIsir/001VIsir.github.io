MySQL 事务隔离级别详解

前言

数据库事务有四个隔离级别，不同隔离级别对并发控制和数据一致性有不同影响。这是面试中常考的知识点，也是在实际开发中必须理解清楚的概念。

四个隔离级别

隔离级别 脏读 不可重复读 幻读
READ UNCOMMITTED 可能 可能 可能
READ COMMITTED 不可能 可能 可能
REPEATABLE READ（默认） 不可能 不可能 可能
SERIALIZABLE 不可能 不可能 不可能

1. READ UNCOMMITTED（读未提交）

最低级别，允许读取其他事务未提交的数据。

问题：会产生脏读 — 读取到其他事务还未提交的数据，如果那个事务最终回滚，读到的就是错误数据。

2. READ COMMITTED（读已提交）

只允许读取其他事务已提交的数据。Oracle、SQL Server 默认是这个级别。

特点：每次 SELECT 都会生成新的 ReadView，所以同一事务内两次相同的查询可能看到不同的数据（其他事务提交了新的修改）。

这就是不可重复读现象。

3. REPEATABLE READ（可重复读）

MySQL InnoDB 默认隔离级别。事务开始时创建 ReadView，整个事务内复用同一个 ReadView，所以同一事务内每次读取结果都一致。

MySQL 额外保证：InnoDB 在 RR 级别下通过 Next-Key Lock 解决了幻读问题（即同一事务内两次查询结果行数相同）。

4. SERIALIZABLE（串行化）

最强隔离级别，所有事务串行执行。读写互相阻塞，完全避免了一切并发问题，但性能最差。

MVCC 在不同隔离级别下的体现

  • READ COMMITTED — 每次读取都生成新的 ReadView
  • REPEATABLE READ — 事务开始时生成 ReadView，事务内不改变

这就是两种隔离级别在表现上不同的根本原因。

如何选择隔离级别

场景 推荐级别
一般业务 REPEATABLE READ（默认）
金融、账务类 SERIALIZABLE 或代码层面加锁
报表、数据分析 READ COMMITTED
高并发读多写少 READ COMMITTED

Spring 中设置隔离级别

总结

四种隔离级别从低到高：读未提交 → 读已提交 → 可重复读 → 串行化。级别越高，数据越安全，但性能越差。MySQL InnoDB 默认使用 REPEATABLE READ，并通过 MVCC + Next-Key Lock 在保证安全的同时兼顾性能。
