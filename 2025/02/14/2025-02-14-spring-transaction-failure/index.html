@Transactional 失效的几种场景

前言

Spring 的 @Transactional 是我们最常用的声明式事务控制，但很多时候它会悄悄失效，导致数据不一致。了解哪些场景会导致事务失效，是写出正确业务代码的基础。

1. 方法内部调用（自调用）

这是最常见的事务失效原因。

this.pay() 是直接调用，不是通过 Spring 代理对象，所以代理增强不生效。

解决方案：

  • 把 pay() 拆到另一个 Service 中调用
  • 注入自身代理对象 ((OrderService)AopContext.currentProxy()).pay(order)

4. 传播行为配置错误

NOT_SUPPORTED、NEVER 等传播行为会导致事务不生效。

5. 多数据源环境下没有指定 TransactionManager

6. 访问修饰符不是 public

Spring 事务是基于 AOP 代理实现的，private 方法无法被代理增强。

7. 同一个类中两个方法互相调用

跟第 1 点一样，通过 this.xxx() 调用不走 Spring 代理。

8. 数据库本身不支持事务

MySQL 的 MyISAM 引擎不支持事务，即使加了 @Transactional 也不会生效。InnoDB 才支持事务。

最佳实践

  1. @Transactional 只能加在 public 方法上
  2. 同一个类中避免自调用，需要调用时注入自身
  3. 显式指定 rollbackFor = Exception.class
  4. 避免在事务方法中执行耗时操作（RPC、批量插入等）
  5. 养成用单元测试验证事务是否生效的习惯

总结

Spring 事务失效的原因大多跟 Spring 代理机制有关：非 public 方法、自调用、异常被吞、异常类型不匹配、传播行为错误等。理解这些场景，能帮我们避免很多生产环境的数据一致性问题。
