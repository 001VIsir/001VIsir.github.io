垃圾回收算法详解

前言

Java 程序的内存管理是自动化的，开发者不需要手动释放内存，这得益于 JVM 的垃圾回收（Garbage Collection）机制。但理解 GC 算法，能帮助我们写出更高效的程序，也能更好地排查 OOM 问题。

如何判断对象已”死”？

GC 在回收对象之前，需要先判断哪些对象是”垃圾”（不再被使用的对象）。

引用计数法

每个对象有一个引用计数器，有地方引用就 +1，引用失效就 -1。计数器为 0 就是垃圾。

问题：无法解决循环引用（A 引用 B，B 引用 A，但没有任何其他对象引用它们）。

可达性分析（主流）

从 GC Root 开始，通过引用链向下搜索，所有 GC Root 不可达的对象就是垃圾。

可以作为 GC Root 的对象：

  • 虚拟机栈（栈帧中的局部变量表）中引用的对象
  • 方法区中类静态属性引用的对象
  • 方法区中常量引用的对象
  • 本地方法栈中 JNI 引用的对象
  • JVM 内部引用（Class 对象、异常对象等）
  • 同步锁（synchronized）持有的对象

垃圾回收算法

1. 标记-清除（Mark-Sweep）

缺点：

  • 效率不稳定，标记和清除都需要扫描全堆
  • 会产生内存碎片，分配大对象时可能提前触发下一次 GC

2. 复制算法（Copying）

将内存分成两块，每次只使用一块。GC 时把存活对象复制到另一块，然后直接清除整块内存。

现代 JVM 把堆分为一个 Eden区和两个 Survivor 区（8:1:1），使用 minor GC。Eden 区满时，把存活对象复制到 Survivor 区。

优点：没有内存碎片，实现简单高效。
缺点：可用内存降为原来的一半。

3. 标记-整理（Mark-Compact）

在标记-清除的基础上，增加一步”整理”：让存活对象向一端移动，然后清理边界以外的内存。

优点：没有内存碎片，适合老年代。
缺点：移动存活对象需要更新引用，成本较高。

分代收集

现代 JVM 根据对象存活周期采用分代收集：

常见 GC 收集器

收集器 作用于 算法 特点
Serial 新生代 复制 单线程，最古老
ParNew 新生代 复制 Serial 多线程版
Parallel Scavenge 新生代 复制 追求吞吐量
Serial Old 老年代 标记-整理 单线程
Parallel Old 老年代 标记-整理 多线程，吞吐量优先
CMS 老年代 标记-清除 追求低停顿
G1 全堆 标记-整理+复制 可预测停顿
ZGC 全堆 标记-整理 停顿时间 < 1ms

G1 和 ZGC 的区别

G1 把堆分成多个大小相等的 Region，通过追踪每个 Region 的回收价值，优先回收价值最高的 Region。ZGC 则进一步优化，停顿时间控制在毫秒级以内，且不依赖分代。

JDK 11+ 推荐优先尝试 ZGC 或 G1。

总结

垃圾回收的核心问题是判断对象生死和高效回收。JVM 通过可达性分析找到垃圾，复制算法处理新生代，标记-整理处理老年代。理解这些底层原理，能帮助我们写出更合理的大对象分配策略，减少 GC 频率，提升系统性能。
