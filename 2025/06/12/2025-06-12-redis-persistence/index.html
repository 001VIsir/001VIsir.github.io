Redis RDB 和 AOF 持久化详解

前言

Redis 数据都在内存中，如果突然宕机或断电，数据就会丢失。Redis 提供了两种持久化机制来保证数据不丢失：RDB 和 AOF。

1. RDB（Redis Database）

RDB 是定时生成整个数据库的快照，保存为一个 .rdb 二进制文件。

触发方式

优点

  • 恢复数据极快（直接加载 .rdb 文件）
  • 文件紧凑（是某个时刻的完整快照）
  • 适合做灾备、冷备份

缺点

  • 可能会丢失最后一次快照之后的数据（取决于触发时机）
  • BGSAVE 时 fork() 子进程，有内存和 CPU 开销
  • 如果数据量很大，fork 可能造成服务短暂停顿

工作原理

这个过程是 Copy-On-Write 的，子进程共享主进程内存，只有数据变化时才复制。

2. AOF（Append Only File）

AOF 记录所有写命令，追加到 .aof 文件末尾。恢复时重放所有命令。

配置

三种同步策略

策略 含义 性能 数据安全性
always 每条命令都同步 最慢 最高，几乎不丢数据
everysec 每秒同步一次 中等 最多丢 1 秒数据
no 操作系统决定 最快 丢数据最多

AOF 重写机制

随着时间推移，AOF 文件会越来越大。Redis 提供重写机制压缩文件：

原理：遍历当前数据，用最少的命令重建数据状态（而不是记录所有历史操作）。

当 AOF 文件体积达到上次重写后的 100% 时，会自动触发重写（可配置 auto-aof-rewrite-percentage）。

3. RDB 和 AOF 对比

特性 RDB AOF
文件格式 二进制 文本命令
恢复速度 快 慢（需重放命令）
数据完整性 可能有几秒数据丢失 可配置（everysec 丢 1 秒）
文件大小 小（完整快照） 大（累积所有命令）
资源消耗 fork 子进程 持续写磁盘
启动顺序 先加载 RDB 先加载 AOF（优先）

4. 生产环境怎么选？

同时开启两者（推荐）：

Redis 启动时优先加载 AOF（因为 AOF 数据更完整），如果 AOF 不存在才加载 RDB。

AOF + everysec 适合对数据安全性有一定要求但不想性能损失太大的场景。

纯缓存场景 — 如果 Redis 只用作缓存、不需要持久化，可以关闭所有持久化。

总结

RDB 是定时快照，恢复快但可能丢数据；AOF 是持续记录命令，安全性更高但文件更大。生产环境建议同时开启两者，用 AOF 保证数据完整性，用 RDB 做快速灾备恢复。
