Java 动态代理原理

前言

Spring AOP、MyBatis 映射器、Hibernate 延迟加载……这些框架都大量使用了动态代理。理解动态代理的原理，是理解这些框架底层机制的关键。

什么是代理模式？

代理模式为其他对象提供一种替身，以控制对这个对象的访问。

静态代理的缺点：每个类都要写一个代理类，类爆炸。

JDK 动态代理

JDK 动态代理要求被代理的类实现接口。

JDK 动态代理原理

当调用 proxy.create(user) 时，实际上是调用了 InvocationHandler.invoke() 方法，在调用前后可以加增强逻辑。整个过程是运行时通过反射生成字节码，创建一个 $Proxy0 类继承 Proxy、实现 UserService 接口。

CGLib 动态代理

CGLib 通过继承的方式代理，不需要接口，效率更高（因为使用 FastClass）。

JDK 代理 vs CGLib 代理

特性 JDK 动态代理 CGLib 代理
实现方式 实现接口 继承父类
依赖 JDK 内置 第三方库（spring-core 自带）
执行效率 反射调用，稍低 生成 FastClass，效率更高
代理范围 只能代理实现了接口的类 可以代理任意类
限制 无法代理没有接口的类 无法代理 final 类和 final 方法

Spring AOP 中的选择

Spring AOP 默认：

  • 如果目标类有接口，用 JDK 动态代理
  • 如果目标类没有接口，用 CGLib 代理
  • 可以通过配置 proxy-target-class="true" 强制都使用 CGLib

代理模式的应用场景

  1. Spring AOP — 事务管理（@Transactional）、日志记录（@Around）
  2. MyBatis — Mapper 代理，JDBC 操作的拦截
  3. Hibernate — 延迟加载（懒加载）
  4. Spring Data JPA — 实体变更追踪
  5. 全局异常处理 — 统一封装异常返回格式

总结

动态代理让程序在运行时动态生成代理类，省去了静态编写代理类的麻烦。JDK 代理基于接口，需要反射调用；CGLib 基于继承，效率更高但无法代理 final 类。Spring 和 MyBatis 正是利用这一特性，实现了声明式事务、Mapper 代理等强大功能。
