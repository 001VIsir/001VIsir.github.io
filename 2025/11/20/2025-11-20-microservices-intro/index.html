微服务架构入门：什么是微服务

前言

微服务是近年来后端领域最火的概念之一。很多团队喊着要”微服务化”，但实际上对微服务的理解还停留在”把大项目拆成小项目”的层面。这篇文章聊聊微服务到底是什么，以及它解决了什么问题。

什么是微服务？

维基百科的定义：微服务是一种软件架构风格，以一个个小而自治的服务来构建应用，每个服务运行在独立进程中，服务之间通过轻量级通信机制（HTTP REST、gRPC）协作。

简单来说：把一个大应用拆成多个独立的小应用，每个小应用专注做好一件事。

微服务 vs 单体架构

单体架构

所有模块打包在一个 jar / war 中：

  • 用户模块、订单模块、支付模块、商品模块……都在同一个进程里
  • 优点：简单，部署容易，本地开发调试方便
  • 缺点：一个模块出问题拖垮整个应用；所有代码耦合在一起，改动风险大

微服务架构

每个模块独立部署为服务：

  • 用户服务、订单服务、支付服务、商品服务……各自独立进程
  • 通过 HTTP / gRPC / 消息队列通信
  • 优点：独立部署、故障隔离、技术异构、按需扩展
  • 缺点：复杂度高、运维难度大、数据一致性挑战

微服务的核心特征

独立部署 — 每个服务可以独立打包、部署、升级，不影响其他服务。

按业务边界拆分 — 按业务领域划分服务，而不是按技术层次划分（不是把 Controller 放一个服务、DAO 放另一个服务）。

去中心化 — 每个服务自己管理数据库，不共享数据库。

进程隔离 — 每个服务运行在独立的进程（容器）中。

技术多样性 — 不同服务可以用不同的技术栈，只要对外接口一致。

微服务带来的问题

微服务不是银弹，它解决了单体架构的一些问题，但也引入了新的复杂度：

分布式事务 — 不同服务有自己的数据库，一个业务操作可能涉及多个服务的数据一致性。解决方案：Saga 模式、TCC 模式、本地消息表。

服务间通信 — HTTP REST 调用可能超时、重试、失败，需要考虑幂等性、重试机制。

服务发现 — 服务实例动态上下线，如何让调用方找到服务？解决方案：Eureka、Nacos、Consul。

分布式链路追踪 — 一个请求经过多个服务，如何追踪全链路？解决方案：Skywalking、Zipkin、Jaeger。

运维复杂度 — 10 个服务怎么部署、监控、日志聚合？需要容器化（Docker + Kubernetes）和完善的 DevOps 体系。

什么时候该用微服务？

适合微服务的场景：

  • 团队规模较大（10 人以上），不同团队负责不同服务
  • 业务复杂度高，需要独立演进
  • 对可用性和伸缩性要求高

不适合微服务的场景：

  • 团队小（3-5 人以下）
  • 业务复杂度低，团队经验不足
  • 还没有搞定 CI/CD、容器化等基础设施

从单体开始，逐步拆分 — 这是推荐的做法。先把单体架构做好，等业务复杂到不得不拆时再拆。

微服务技术栈

领域 常用技术
服务开发 Spring Boot / Go / Node.js
服务网关 Spring Cloud Gateway / Kong / Zuul
服务发现 Nacos / Eureka / Consul
配置中心 Nacos / Apollo / Spring Cloud Config
熔断限流 Sentinel / Hystrix
分布式事务 Seata / Saga
链路追踪 Skywalking / Zipkin / Jaeger
容器编排 Kubernetes / Docker Swarm
持续集成 Jenkins / GitLab CI / Argo CD

总结

微服务是一种架构风格，不是银弹。它通过把大应用拆成小服务来实现独立部署、故障隔离和技术多样性，但同时也带来了分布式系统的固有复杂性。是否采用微服务，应该根据团队规模、业务复杂度和技术储备来综合判断，而不是盲目跟风。
